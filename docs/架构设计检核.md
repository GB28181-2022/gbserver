# 架构设计检核报告

## 一、不确定或需明确的内容

### 1. 点播流程与 ZLM 的先后关系

- **现状**：5.2 写「SIP 向设备发 INVITE → 设备 200 OK + SDP → 流媒体服务建立接收通道」；9.3 写「设备 Register 时本系统调用 openRtpServer → 在 200 OK 的 SDP 中填入端口」。
- **矛盾点**：国标点播一般是**平台先**在 INVITE 的 SDP 里带**媒体接收地址**，设备再往该地址推流。因此应先调用 ZLM `openRtpServer(stream_id)` 得到端口，再发 INVITE 并在 SDP 中携带本机 IP 与端口；而不是设备返回 SDP 后再建通道。
- **建议**：统一为「发起点播时先 openRtpServer → 用返回的 port 生成 SDP → 发 INVITE(SDP) → 设备按 SDP 推流到 ZLM」。

### 2. 「流媒体服务」在本系统中的边界

- **现状**：3.1 将「流媒体服务」列为本系统模块，职责含「接收 RTP、录像存储、回放」；实际 RTP 由 ZLM 接收。
- **不确定**：本系统内「流媒体服务」是否仅指**调用 ZLM HTTP API、协调 openRtpServer/startRecord/播放 URL** 的模块，而不自行收 RTP？
- **建议**：在 3.3 中明确「流媒体服务」= 对 ZLM 的封装与协调（调 API、维护 stream_id 与设备映射、提供录像/播放 URL 给前端），RTP 接收与存储均在 ZLM。

### 3. 客户端模式下的媒体源（已明确）

- **结论**：上级请求的流此时未在流媒体中，**由 SIP 指令控制下级向本系统推流，再转发给上级**。流程：收到上级 INVITE → 本系统 openRtpServer → 向下级发 INVITE（点播/回放）→ 下级向 ZLM 推流 → ZLM 流注册后 startSendRtp 转发上级。
- 已按此更新架构 3.3 媒体推送、5.7、9.4、9.6。

### 4. 白名单 + 独立鉴权密码的优先级

- **现状**：需求有「白名单可免鉴权」「每个下级可配独立鉴权密码」。
- **不确定**：若某设备既在白名单又配置了独立密码，是「免鉴权放行」还是「仍校验密码」？
- **建议**：在配置管理或 6.2 接口中约定规则（例如：白名单且未配置独立密码则免鉴权；若配置了独立密码则校验密码）。

### 5. 前端播放 URL 的获取与访问方式

- **现状**：4.2、9.5 写前端通过 ZLM 的 RTSP/HTTP-FLV/HLS URL 用 jessibuca 播放。
- **不确定**：（1）jessibuca 常用 HTTP-FLV/WebSocket-FLV/HLS，是否以 HTTP-FLV 或 HLS 为主、不直接用 RTSP？（2）播放 URL 由本系统后端从 ZLM 查询后下发给前端，还是前端直连 ZLM？若直连 ZLM，需考虑跨域与 ZLM 地址暴露。
- **建议**：明确主推协议（如 HTTP-FLV 或 HLS）及 URL 下发方式（建议后端提供「获取某通道播放 URL」的 API，后端再调 ZLM 或查映射）。

### 6. 双模式同时启用时的端口与资源

- **现状**：2.1 写服务端与客户端模式「可独立或同时启用」，7.1 写本系统单进程。
- **不确定**：同时启用时，SIP 服务端 5060 与 SIP 客户端所用端口、ZLM 的 openRtpServer 与 startSendRtp 是否共用同一 ZLM 实例，是否有资源/端口冲突说明。
- **建议**：在 7.1 或 7.2 增加一句：双模式同时启用时，SIP 服务端与客户端端口隔离，ZLM 同时承担收流（openRtpServer）与向级联平台推流（startSendRtp）。

## 二、遗漏内容

### 1. ZLM 播放相关端口未在端口规划中列出

- **遗漏**：7.2 仅列 ZLM HTTP API 880/8443、RTP 10000 与 openRtpServer 动态端口；未列 ZLM 对外提供播放的端口（如 RTSP 554、RTMP 1935、HTTP 80 等）。
- **影响**：部署与防火墙规划不完整。
- **建议**：在 7.2 增加 ZLM 播放端口（如 RTSP 554、RTMP 1935、HTTP-FLV/HLS 所用端口），并注明「以 ZLM 实际配置为准」。

### 2. ZLM Web Hook 回调地址

- **遗漏**：9.3 写「通过 ZLM Web Hook（如 on_stream_changed）感知流注册」，未写本系统如何配置 Hook 地址、ZLM 回调本系统的 URL（如 `http://本系统:8080/api/zlm/hook`）。
- **影响**：实现时不知道 Hook 配置与路由。
- **建议**：在 9.3 或 6.2 中增加「本系统向 ZLM 注册 Web Hook 回调地址，用于 on_stream_changed、on_stream_none_reader 等」，并说明由本系统哪一模块提供该 HTTP 接口。

### 3. 录像元数据与历史检索

- **遗漏**：录像存储写由 ZLM startRecord 完成，但「历史检索」按设备、时间、事件等条件检索在架构中未体现；未写录像元数据（设备、通道、时间范围、文件路径或 ZLM 流 ID）存何处（本系统 PostgreSQL 还是仅用 ZLM API 查）。
- **影响**：历史检索、回放列表的实现方式不明确。
- **建议**：在 3.3 流媒体服务或设备管理/告警中增加：录像元数据存 PostgreSQL（设备、通道、时间、ZLM 流 ID 或文件路径），历史检索由本系统查库 + 调 ZLM 获取播放 URL；或明确「仅通过 ZLM API 检索」并说明 ZLM 能力。

### 4. 前后端 REST API 未细化

- **遗漏**：4.3 已标「未细化 REST API」，6.2 仅列模块间接口，无对前端的 API 清单。
- **影响**：前端与后端对接边界不清晰。
- **建议**：在 6 接口设计或单独「前后端接口」小节列出主要 API（如配置 CRUD、设备树、点播/停播、回放、告警列表、PTZ、获取播放 URL 等），可先列清单不写报文，后续设计阶段再补细节。

### 5. 日志与可维护性

- **遗漏**：NFR-004 要求关键操作有日志，架构中无日志模块、日志级别、存储方式（文件/库）、是否落 PostgreSQL。
- **影响**：运维与排障依据不足。
- **建议**：在 3 模块划分或 7 部署中增加「日志」：关键操作（注册、点播、控制、告警、ZLM 调用）写日志，存储方式与级别待实现阶段确定；在技术选型中明确采用 ZLToolKit 提供统一的日志、线程池、定时器等基础设施，上层模块复用该能力。

### 6. 前端访问控制与 API 鉴权

- **遗漏**：前端 8080 是否需登录、会话管理；本系统调用 ZLM 时使用 ZLM 的 secret，但本系统 8080 API 是否需 token/session 未写。
- **影响**：安全与权限边界不清。
- **建议**：在 4.2 或 7/8 中增加一句：前端访问与 API 鉴权策略在实现阶段确定（如登录态、API token 或统一网关）。

### 7. 错误与重试策略

- **遗漏**：SIP 信令失败、ZLM 调用失败（如 openRtpServer 失败、startSendRtp 超时）时的处理与重试未写。
- **影响**：实现时行为不一致。
- **建议**：在 5 数据流或 6 接口中简要说明：关键调用失败时的返回与重试策略在实现阶段约定（如 openRtpServer 失败则 INVITE 4xx、startSendRtp 失败重试次数等）。

### 8. 手动添加与点播的关系

- **现状**：SVR-101 手动添加无需等待主动注册；3.3 写「预置设备/平台信息」。
- **遗漏**：手动添加后若设备尚未注册，点播时是否允许发 INVITE（可能无响应），还是仅展示「离线」不可点播？未写。
- **建议**：在 3.3 设备管理或 5.2 点播流程中约定：仅对已上线（已注册）设备发起点播；手动添加仅预置信息，待设备注册后再可点播。

### 9. 前端 UI 组件库选型

- **现状**：前端技术栈已明确使用 Vue，但 UI 组件库未在检核报告中单独点名。
- **影响**：界面风格与组件选型若不统一，可能导致前端实现阶段出现多个 UI 库混用的问题。
- **建议**：在架构设计的技术选型和前端页面章节中，明确采用 Element Plus 作为 Vue 3 的标准 UI 组件库，用于表单、表格、布局、对话框等管理界面组件，统一交互与样式规范。

## 四、建议的文档修订优先级

1. **高**：点播流程与 openRtpServer 的先后关系（5.2 与 9.3 统一）；ZLM 播放端口补充（7.2）；「流媒体服务」边界（3.3）；ZLM Web Hook 回调（9.3/6.2）。
2. **中**：录像元数据与检索；客户端媒体源；前端播放 URL 获取方式；白名单与独立密码优先级。
3. **低**：REST API 清单；日志与鉴权；错误与重试；手动添加与点播关系。

以上内容可在架构设计正文中按优先级补充或引用本检核报告。
