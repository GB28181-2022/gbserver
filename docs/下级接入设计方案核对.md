# 下级接入设计方案核对

> 目的：在您准备下级国标平台并配置本系统国标信息之前，与您确认「下级接入」的设计与分工，确定后继续实现。  
> 依据：《架构设计》《架构设计检核》《编码阶段就绪检核》、功能清单 SVR-101～110。

---

## 一、角色与目标

| 角色 | 说明 |
|------|------|
| **本系统** | 国标服务器系统，作为 **上级平台（SIP 服务端）**，接收下级注册、心跳、目录，后续可发起点播/回放/控制 |
| **您准备的** | **下级国标平台**（或国标设备），作为 **下级**，向本系统发起 SIP 注册、心跳、上报目录等 |

目标：下级能够成功向本系统注册、保活，本系统管理后台能看到该下级（平台管理）及在线状态；为后续目录、点播、回放打基础。

---

## 二、本系统侧已具备与待实现

### 2.1 已具备（无需您配合）

- **管理端**：平台管理（下级平台列表、手动添加、编辑、删除）、黑白名单与独立鉴权配置、摄像头列表、目录编组、告警、概览等。
- **配置**：系统配置中的「本地国标信息」「流媒体配置」的读写（API + 前端）。
- **数据**：`device_platforms`（下级平台）、`cameras`（摄像头）、`gb_local_config`（本机国标）、`media_config`（流媒体）等表与接口已就绪。

### 2.2 待实现（下级接入核心）

| 序号 | 内容 | 说明 |
|------|------|------|
| 1 | **SIP 信令服务** | 本系统监听 **5060**（可配置）UDP/TCP，接收下级发来的 SIP 消息 |
| 2 | **注册（Register）** | 解析下级 SIP Register，按黑白名单 + 鉴权（含独立密码）校验，通过则写入/更新 `device_platforms` 与在线状态，并回复 200 OK |
| 3 | **心跳（Keepalive）** | 解析心跳 Message，更新该下级 `online`、`last_heartbeat_at`；超时未心跳则置为离线 |
| 4 | **本地国标与信令地址** | 本系统「本地国标信息」供下级配置用：国标 ID、SIP 域、鉴权用户名/密码、传输方式（UDP/TCP）；**信令地址** = 本系统部署所在机器的 IP（或域名），**信令端口** = 5060（若表结构暂无单独字段，则端口固定 5060，地址取本机 IP 或后续在配置中增加「信令 IP」） |

### 2.3 后续阶段（本次不必须）

- 目录（Catalog）：接收下级上报设备/通道树，维护到 `cameras` 等。
- 点播/回放：本系统向已注册下级发 INVITE，协调 ZLM 收流与播放 URL。
- 告警：接收下级 Notify 告警并落库。

## 三、您需要提供的本系统国标信息（给下级配置用）

下级平台在配置「上级」时，需要填写**本系统**的国标相关信息，以便向本系统发起注册与心跳。请在本系统**系统配置 → 本地国标信息**中配置并保存，然后提供给下级使用：

| 配置项 | 含义 | 您在下级侧填写 |
|--------|------|----------------|
| **国标 ID（gb_id）** | 本系统国标 20 位编码 | 与「本地国标信息」中一致 |
| **SIP 域（domain）** | 本系统 SIP 域 | 与「本地国标信息」中一致 |
| **信令地址** | 本系统 SIP 服务监听地址 | **本系统部署机器的 IP 或域名**（与 5060 可达） |
| **信令端口** | 本系统 SIP 服务监听端口 | **5060**（若后续支持配置则以实际配置为准） |
| **传输方式** | UDP 或 TCP | 与「本地国标信息」中一致（UDP/TCP） |
| **鉴权用户名** | 本系统侧用于 Digest 鉴权的用户名 | 与「本地国标信息」中一致 |
| **鉴权密码** | 本系统侧用于 Digest 鉴权的密码 | 与「本地国标信息」中一致 |

说明：当前「本地国标信息」接口中有 `gbId`、`domain`、`username`、`password`、`transport`；**信令端口**在实现时固定为 5060，**信令地址**为运行本系统的机器 IP（或后续在配置中增加「信令 IP」字段）。下级配置时，**上级信令 IP** = 本机 IP，**上级信令端口** = 5060。

## 四、您需要做的准备工作（下级侧）

1. **准备一台国标下级平台**（或设备），支持作为 GB28181 下级向上级注册、发心跳。
2. **网络可达**：确保该下级能访问本系统部署机器的 **5060 端口**（UDP 和/或 TCP，与传输方式一致）。
3. **在下级侧配置「上级」**：
   - 上级国标 ID、SIP 域、信令 IP、信令端口 5060、传输方式、鉴权用户名/密码，均按上表从本系统「本地国标信息」及实际部署得到。
4. **下级国标 ID**：若本系统启用了白名单，请将下级的国标 ID 加入白名单，或在「平台管理」中先**手动添加**该下级（gb_id 与下级一致），并视需求配置独立鉴权密码。

## 五、设计要点确认（请逐项确认）

以下为《架构设计》与检核报告中的关键约定，请确认是否同意作为实现依据。

| 序号 | 要点 | 约定 |
|------|------|------|
| 1 | **鉴权顺序** | 先判黑名单（命中则拒绝）→ 再判白名单；白名单下若该下级配置了独立鉴权密码则校验密码，未配置则免鉴权。 |
| 2 | **手动添加与注册** | 支持手动添加下级（预填 gb_id、名称等）；下级首次 Register 通过后，与已有记录按 gb_id 匹配并更新在线状态；未手动添加也可在注册时自动创建设备平台记录（若需求允许）。 |
| 3 | **点播与在线** | 仅对**已上线**（已注册且心跳正常）的下级/通道发起点播；手动添加但尚未注册的显示离线，不可点播。 |
| 4 | **信令与媒体** | 信令（SIP）由本系统 5060 处理；媒体（RTP）由 ZLMediaKit 接收，本系统只做协调（openRtpServer 等），点播/回放阶段再实现。 |
| 5 | **端口** | 本系统 SIP 服务端：**5060**（UDP/TCP 依配置）；ZLM 与 8080 等按现有架构不变。 |
| 6 | **心跳超时** | 若在约定时间内未收到下级心跳，将该下级标记为离线；具体超时时间（如 2 倍心跳间隔）实现时与您约定或取合理默认值。 |

## 六、实现顺序建议（本系统侧）

1. **SIP 服务端**：基于 PJSIP（或现有选型）监听 5060，接收 UDP/TCP。
2. **Register 处理**：解析 Register，黑白名单 + 鉴权，写库 `device_platforms`（及可选 `cameras`），回 200 OK / 4xx。
3. **Keepalive 处理**：解析心跳 Message，更新 `online`、`last_heartbeat_at`；定时任务将超时未心跳的下级置为离线。
4. **前端**：平台管理列表已存在，仅需确保「在线/离线」来自库表并随心跳与超时更新。
