## 二、当前实现状态（本系统）

### 2.1 已完成

- **基础设施**：C++ 工程、ZLToolKit、8080 HTTP、PostgreSQL、Vue 前端 + Element Plus。
- **用户与鉴权**：AUTH-001～003（登录/登出/修改密码、token、未登录 401）。
- **配置管理**：CFG-001/002（本地国标、流媒体 GET/PUT，API+前端）。
- **平台管理（下级）**：SVR-101 相关（device_platforms 列表、手动添加、编辑、删除、黑白名单/独立鉴权配置、摄像头列表、目录编组、告警、概览）。
- **上级平台**：平台接入（upstream_platforms CRUD、心跳间隔），前端已接 API。
- **告警**：SVR-501/502（接收、确认、处置、记录，API+前端）。
- **录像检索**：GET /api/replay/segments，前端回放页接入。
- **其他**：健康检查、REST 约定、库表（users、gb_local_config、media_config、device_platforms、cameras、catalog、alarms、replay 等）。

### 2.2 未完成（与下级接入及后续相关）

- **SIP 信令服务**：未监听 5060，未接入 PJSIP。
- **SVR-106 设备注册**：未解析 SIP Register，未做黑白名单与鉴权、未写 device_platforms/在线状态。
- **SVR-110 心跳保活**：未解析心跳 Message，未更新 online/last_heartbeat_at，无超时置离线。
- **SVR-107 设备注销**：未处理 SIP 注销或超时离线。
- **SVR-108 设备目录**：未接收/解析 Catalog，未维护 cameras 等。
- **SVR-201/303 点播/回放**：未与 ZLM 协调、未发 INVITE，无播放 URL 接口。
- **SVR-401 PTZ、SVR-501 告警 Notify**：告警已支持 HTTP 上报，SIP Notify 未接。

## 三、推进步骤（分阶段，确认后执行）

### 阶段 0：前置（您侧 + 本系统配置）

- 您：准备下级国标平台，保证其能访问本系统部署机 **5060**（UDP/TCP）。
- 本系统：在「系统配置 → 本地国标信息」中配置并保存 **国标 ID、SIP 域、鉴权用户名/密码、传输方式**；信令端口固定 **5060**，信令地址为部署机 IP。
- 若启用白名单：在平台管理中**手动添加**该下级（gb_id 与下级一致），或将下级国标 ID 加入白名单；如需鉴权则配置独立密码。

**验收**：配置可保存并可被前端/API 读取；您侧可在下级平台配置「上级」为本系统（IP、5060、域、国标 ID、鉴权信息）。

---

### 阶段 1：SIP 服务端与注册、心跳（SVR-106、SVR-110，部分 SVR-107）

**目标**：本系统监听 5060，接收下级 SIP Register 与心跳，完成鉴权与在线状态维护，管理端可见下级在线/离线。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 1.1 | **引入 PJSIP 或等效 SIP 栈** | 工程可编译、可创建 SIP 服务端、监听 5060（UDP 优先，可选 TCP） |
| 1.2 | **SIP 服务端进程内启动** | main 或统一入口中在 8080 之外启动 SIP 监听（端口可配置，默认 5060），与现有 HTTP 同进程 |
| 1.3 | **解析 Register** | 从 SIP Register 中解析设备/平台国标 ID、鉴权信息（Digest）；查询 gb_local_config、device_platforms（黑白名单、独立密码），按《下级接入设计方案核对》鉴权顺序校验 |
| 1.4 | **注册通过后写库** | 通过则插入/更新 device_platforms（gb_id、名称等可从 Register 取或用手动添加的），置 online=true，回 200 OK；拒绝则回 4xx |
| 1.5 | **解析心跳 Message** | 按 GB28181 解析心跳 SIP Message，识别设备/平台，更新对应 device_platforms 的 online、last_heartbeat_at（或等效字段） |
| 1.6 | **心跳超时置离线** | 定时任务（如每分钟）检查 last_heartbeat_at，超过阈值（如 2 倍心跳间隔或固定 120s）将 online 置为 false |
| 1.7 | **（可选）注销** | 收到 SIP 注销或连接断开时，将对应下级 online 置为 false（可放在 1.4/1.5 同批实现） |

**验收**：下级向本系统 5060 发起注册与心跳后，平台管理列表中该下级显示在线；停止心跳或超时后显示离线。

**依赖**：本地国标与黑白名单/独立密码已配置；您侧下级已配置本系统为上级并开启注册与心跳。

---

### 阶段 2：设备目录（SVR-108，可选与阶段 1 合并）

**目标**：接收下级上报的 Catalog（设备/通道树），解析并写入 cameras 等，支持按平台/设备查询。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 2.1 | **解析 Catalog Message** | 识别 SIP Message 中的 Catalog XML/ body，解析设备与通道列表、国标 ID、名称、上下级关系 |
| 2.2 | **与 device_platforms 关联** | 按平台国标 ID 关联到已注册的 device_platforms，写入或更新 cameras（id、name、platform_id 等） |
| 2.3 | **去重与更新** | 同一通道多次上报时更新而非重复插入；可设「目录来源=该下级」便于后续扩展 |

**验收**：下级上报目录后，本系统「摄像头管理」或设备树中能展示该下级下的通道列表。

**依赖**：阶段 1 已完成，下级支持上报 Catalog。

---

### 阶段 3：实时点播（SVR-201）与播放 URL

**目标**：对已在线下级/通道发起点播，本系统协调 ZLM 收流，前端可获取播放地址并播放。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 3.1 | **ZLM 集成** | 本系统可调用 ZLM HTTP API（openRtpServer、closeRtpServer 等）；ZLM 已部署，地址/端口可配置（如 media_config 或环境变量） |
| 3.2 | **点播流程** | 应用发起点播 → 本系统调用 ZLM openRtpServer(stream_id) 得端口 → 生成 SDP（本机 IP + 端口）→ 向该通道发 SIP INVITE（点播），SDP 带给下级 → 下级按 SDP 向 ZLM 推 RTP |
| 3.3 | **播放 URL 接口** | 提供 GET /api/stream/play-url 或等价接口，参数为通道/流标识，返回 HTTP-FLV 或 HLS URL（由本系统查 ZLM 或映射得到），供前端 jessibuca 播放 |
| 3.4 | **前端** | 摄像头管理/监控页「预览」调用播放 URL 接口并播放；仅对在线通道可点播（灰显或禁用离线） |

**验收**：选择已在线通道点播后，前端能拉流并播放；无人观看时可后续接 on_stream_none_reader 断流（见架构 9.5）。

**依赖**：阶段 1（可选阶段 2）；ZLM 已部署；本地国标/流媒体配置含本机 IP 与 ZLM 地址。

---

### 阶段 4：历史回放（SVR-303）与录像（SVR-301、SVR-302）

**目标**：支持按通道与时间检索录像、发起回放 INVITE 或使用 ZLM 录像文件，前端可播放。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 4.1 | **录像存储** | 点播或收流后调用 ZLM startRecord；录像元数据（通道、时间范围、流 ID/文件路径）写入本系统库（已有 replay_tasks/replay_segments 可扩展或复用） |
| 4.2 | **历史检索** | 已有 GET /api/replay/segments；可扩展为按设备/事件等条件，或与 ZLM 录像列表对接 |
| 4.3 | **回放流程** | 应用发起回放请求（通道+时间）→ 本系统 openRtpServer → 向该通道发 INVITE（回放，带时间范围），SDP 带给下级 → 下级向 ZLM 推回放流；或对已录像直接返回 ZLM 录像 URL |
| 4.4 | **前端** | 回放页已接 segments 接口；选择时间段与通道后发起回放并播放 |

**验收**：能按通道与时间检索到录像/回放流，前端可播放。

**依赖**：阶段 3；ZLM 录像与 API 可用。

---

### 阶段 5：告警 Notify（SIP）、PTZ（SVR-401）、其他 P0

**目标**：SIP 通道接收下级 Notify 告警并落库（与现有 HTTP 告警统一）；支持 PTZ 控制；按功能清单补齐其余 P0。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 5.1 | **SIP Notify 告警** | 解析 SIP Message 中的 Notify（告警 XML），写入 alarms 表，与现有告警列表/确认/处置共用 |
| 5.2 | **PTZ** | 前端发起 PTZ 指令 → 本系统向指定通道发 SIP Message（PTZ 控制），下级执行 |
| 5.3 | **其他** | 按清单查漏（如设备注销的完整处理、日志与错误策略等） |

**验收**：下级通过 SIP 上报告警可在告警页展示并处理；PTZ 指令可下发并生效（视下级支持情况）。

## 四、建议的近期执行顺序

1. **先确认阶段 0**：您侧准备下级、本系统配置好本地国标；确认《下级接入设计方案核对》中设计要点无异议。
2. **优先做阶段 1**：SIP 服务端 + Register + 心跳 + 超时离线；实现后与您联调，确保管理端可见下级在线/离线。
3. **再定阶段 2～5 顺序**：阶段 2（目录）可与 1 一起做或紧接 1；阶段 3（点播）依赖 1（及可选 2）和 ZLM；阶段 4 依赖 3；阶段 5 可与 3/4 并行或稍后。

## 五、请您确认

请确认以下两项：

1. **推进步骤是否同意**：上述「阶段 0～5」的划分与顺序是否采纳；若有阶段希望提前/延后/合并，请说明。
2. **阶段 1 范围是否锁定**：是否同意先只做「SIP 监听 5060 + Register 鉴权与写库 + 心跳更新在线 + 超时置离线」，目录（Catalog）放在阶段 2 再做。

确认后即按阶段 1 开始实现（PJSIP 引入、5060 监听、Register/心跳解析与设备管理/配置管理对接）。
